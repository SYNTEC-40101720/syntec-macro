# v2.10.0 发布规划

## 版本定位

当前已发布版本为 `v2.9.0`。工作区中的未发布改动新增 Workspace Symbol、静态宏调用引用查找、诊断文档同步检查与 CI，属于向后兼容的用户可见功能，建议作为 `v2.10.0` 发布，不继续追加到 `v2.9.0`。

本规划用于收口当前开发成果。发布前不再扩展新的语法诊断或导航类型，只处理性能、自动化、打包和版本一致性问题。

## 当前基线

已完成：

- Workspace Symbol 可检索静态 G/O 程序、`%@MACRO` 与 N 标签。
- Reference Provider 可查找 `G65/G66/G66.1`、`M98/M198` 静态调用。
- `P#变量`、表达式、注释和普通字符串不会产生静态引用。
- `npm run docs:diagnostics:check` 已接入 `npm test`。
- main 分支与 Pull Request 的基础 CI 已建立。
- `test-demo.nc` 当前保持零诊断。
- ESLint、148 条 validator 测试、36 条扩展模块测试和 VS Code 集成测试均已通过。

## 发布阻断项（P0）

### 1. 导航扫描性能与边界

状态：已完成。

当前 Workspace Symbol 与 References 每次请求仍会枚举工作区并读取候选文件，但已不再批量打开文档。发布前必须继续测量真实 I/O，再决定是否引入缓存，避免大型加工程序目录中出现明显卡顿。

当前进度：

- 已改为通过 `workspace.fs.readFile` 建立文本快照，不再批量 `openTextDocument`。
- 已保留已打开且未保存文档的当前文本优先级。
- 已将单文件宏判定、程序名、符号和调用关系收敛为一次索引构建。
- 已修复 Windows 路径在 Ubuntu CI 环境中的程序名识别。
- 500 文件、20,000 行解析基准：首次 `24.96 ms`，重复 `18.76 ms`。
- 集成测试已确认 Workspace Symbol 查询不会打开额外候选文档。
- 已增加按文档 version 或文件 mtime+size 失效的轻量索引缓存。
- 500 文件、20,000 行真实工作区 I/O 基准：首次 `938 ms`，重复 `394 ms`。
- 已验证文件修改后缓存刷新、文件删除后缓存清理。
- 已验证扫描取消后不读取后续文件，也不返回半成品索引。
- 已验证双根工作区跨根检索和结果去重。

执行项：

- 建立包含至少 500 个候选宏文件、20,000 行宏代码的临时测试工作区。
- 测量首次与重复执行 Workspace Symbol、References 的耗时。
- 验证取消令牌生效，取消后不继续打开剩余文件。
- 验证多根工作区不会漏查或重复返回结果。
- 收紧文件 glob，避免先枚举 `**/*` 再过滤所有无关文件。
- 若重复查询仍明显重复读取，增加按 URI、mtime 或文档版本失效的轻量索引缓存。

验收标准：

- 500 文件基准下首次查询不超过 2 秒，重复查询不超过 500 毫秒。
- 查询期间扩展宿主无持续高 CPU，取消操作可及时结束。
- 现有静态/动态边界集成测试继续通过。

### 2. 发布一致性自动检查

状态：已完成。

此前缺少自动机制检查 package、README、CHANGELOG 与 tag 是否一致，release workflow 可能在版本信息不匹配时仍进入打包。

完成结果：

- 新增 `npm run check:release`，常规检查 package 与 lockfile 两处版本。
- 发布模式支持 `--tag vX.Y.Z`，额外检查 package、README badge 与带日期的 CHANGELOG 版本段。
- package 版本必须使用 `X.Y.Z` 格式，tag 必须使用 `vX.Y.Z` 格式。
- 匹配、lockfile 双字段冲突、版本格式、tag 冲突、README 和 CHANGELOG 缺失均有单元测试。
- `npm test` 已接入常规模式；release workflow 已在测试和打包前接入目标 tag 模式。
- 已验证错误 tag 返回非零状态并输出各文件的具体冲突。

执行项：

- 新增 release consistency 脚本。
- 常规模式检查 `package.json` 与 `package-lock.json` 版本一致。
- 发布模式额外检查 README version badge、CHANGELOG 版本段和目标 tag 一致。
- 在 release workflow 的打包步骤前执行发布模式检查。
- 为版本匹配与不匹配场景增加 Node 单元测试。

验收标准：

- 任一版本源不一致时命令返回非零状态并指出具体文件。
- `v2.10.0` tag 只能打包 `2.10.0` package，并要求 CHANGELOG 存在 `[2.10.0]` 段落。

### 3. CI 覆盖 VS Code 集成测试

状态：实现已完成，待推送后验证 GitHub Actions 实际运行结果。

此前 CI 只运行 `npm test` 与 ESLint，Workspace Symbol 和 References 的真实宿主行为仅在本地验证。

- Ubuntu runner 使用 `xvfb-run -a npm run test:integration` 启动 Electron。

- 集成测试失败时上传 `.vscode-test/user-data/logs/**` 与 `.vscode-test/**/*.log`，保留 7 天。
- 本地普通双根集成套件继续通过。
- 性能门槛基准保持为本地发布前检查，不放入共享 CI runner，避免机器负载导致误报。

执行项：

- 在 GitHub Actions 增加 integration job。
- Linux runner 使用 `xvfb-run -a npm run test:integration`，或使用能够稳定启动 Electron 的等效方案。
- 上传失败日志，便于定位 Extension Host 启动问题。
- release workflow 在打包前执行同一集成测试。

验收标准：

- main push、Pull Request 和 tag release 都会执行 VS Code 集成测试。
- CI 中 Workspace Symbol、References、Definition、Quick Fix 用例全部通过。

## 发布执行（P1）

1. 推送功能与自动化提交，等待 main CI 全部通过。
2. 完成 `chore: release v2.10.0` 并再次等待 CI。
3. 创建并推送 `v2.10.0` tag。
4. 等待 release workflow 生成 `syntec-macro-2.10.0.vsix` 和 GitHub Release。
5. 核对 Release Notes、VSIX 文件名、大小、SHA-256 与 tag。
6. 从 GitHub Release 下载并安装一次发布产物，不使用本地包代替。
7. 确认 `git status --short` 无输出，远端 tag 可查询。

## 发布后观察（P2）

- 观察大型工作区中的 Workspace Symbol 首次和重复查询耗时。
- 观察 Reference Provider 是否需要支持 `includePath` 外部目录；默认仍以工作区引用为边界。
- 收集多根工作区、网络磁盘和大量无扩展名文件场景反馈。
- 若索引缓存进入本版本，观察文件新增、删除、重命名后的失效正确性。
- 发布后只处理回归和阻断问题，新的导航类型进入下一版本规划。

## 非目标

- 不解析 `P#变量`、运行期表达式或动态字符串。
- 不做调用图、Call Hierarchy 或 Rename Provider。
- 不扩大 formatter 的语句重写能力。
- 不在本次发布继续新增诊断规则。
