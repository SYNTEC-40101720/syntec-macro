# v2.9.0 规划

v2.9.0 目标是把 v2.8.x 已完成的诊断体系能力整理成更完整的用户体验，而不是继续零散增加诊断规则。除紧急修复外，v2.8.x 之后建议暂停连续 patch release，改为围绕以下主题积累并一次性发布。

## 发布目标

- 让用户更容易理解诊断结果、Quick Fix 边界和修复路径。
- 提升复杂宏程序的导航与跨文件理解能力。
- 降低诊断规则维护成本，保证 code、文档和测试长期同步。

## 非目标

- 不做运行期表达式求值，例如 `G65 P#1`、动态字符串宏名、运行期计算 GOTO 目标。
- 不做会改变加工语义的自动修复，例如自动猜测 `REPEAT` 的 `UNTIL` 条件、自动选择机器人互斥参数中的某一项。
- 不把 formatter 从保守缩进工具扩展成语句重写器。

## Milestone 1: 诊断体验整合

### 范围

- 在 README 增加“诊断与 Quick Fix”小节，说明：
  - error / warning 的基本含义。
  - 自动 Quick Fix 与说明型 CodeAction 的边界。
  - 完整规则表位于 `docs/诊断规则与修复动作.md`。
- 梳理用户常见问题：
  - 为什么控制结构开头行不能加 `;`。
  - 为什么 `END_*` 和完整语句必须加 `;`。
  - 为什么命名变量、FANUC 比较关键字、机器人直接引数 `=` 会报错。
- 让诊断文档更面向用户：保留 code 表，同时增加按场景分组的短说明。

### 验收标准

- README 中用户能在 1 次跳转内找到诊断规则与 Quick Fix 说明。
- `docs/诊断规则与修复动作.md` 继续可由 `npm run docs:diagnostics` 生成。
- `npm test` 与 `npm run test:integration` 通过。

## Milestone 2: 复杂宏程序导航

### 范围

- 增强静态导航体验：
  - 评估 workspace symbol，列出 `%@MACRO`、`N` 标签、静态可解析的 G/O 宏程序入口。
  - 评估宏调用引用视图，基于静态 `G65/G66/G66.1 P_` 与 `M98/M198 P_` 关系。
- 保持当前动态边界：
  - 动态变量、表达式或运行期生成目标不做跳转。
  - 对动态目标可考虑说明型提示，而不是错误诊断。

### 验收标准

- 静态可解析调用目标继续准确跳转。
- 动态目标不误跳、不误报。
- 新增导航能力具备 VS Code 集成测试覆盖。

## Milestone 3: 诊断规则维护自动化

### 范围

- 增加诊断文档同步检查脚本，例如：
  - 生成文档到临时内容。
  - 与仓库中的 `docs/诊断规则与修复动作.md` 对比。
  - 不一致时退出非零状态。
- 将 release 前检查流程收敛为固定命令序列：
  - `npm run docs:diagnostics`
  - `npm test`
  - `npm run test:integration`
  - `npm run package`
- 评估是否加入 CI，至少覆盖测试和诊断文档同步。

### 验收标准

- 修改 `diagnosticCodes` 或 `diagnosticActions` 后，未同步文档会被脚本发现。
- README / ROADMAP / CHANGELOG 的发布说明与版本号保持一致。
- VSIX 内容包含必要文档，不包含旧版本包。
