## 2. 已核实语法与时序边界

### 2.1 `G65`、`G66`、`G66.1`

- `G65 P_ L_` 是单一宏程序调用；传入字母引数会映射到目标宏的 `#1` 至 `#26`，扩充引数须通过 `GETARG` 读取。
- `G66 P_ L_` 是模式宏调用，后续每个移动单节结束后触发指定程序，直至 `G67` 取消。
- `G66.1 P_ L_` 与 `G66` 共用模式宏的变量生命周期，但在每个单节结束后触发，不限于移动单节。
- `G65`、`G66`、`G66.1` 必须是该行最后一个 G 码。此规则已记录为语法资料，是否已有/应新增诊断须在代码审查后决定。
- `G66` 建立的变量空间不与发出 `G66/G66.1` 的当前程序层共享，只在 `P` 指定的模式宏内共享，至 `G67` 回收。

### 2.2 `M98`、`M198`、`M99`

- `M98 P_ H_ L_` 调用子程序，`H` 省略时从文件开始执行；`P` 省略仅在记忆运转或 MDI 情境指向程序自身。
- `M198 P_ H_ L_` 必须有 `P`；缺失时会触发 `COR-052`。它只可调用档案型式副程序。
- `M198` 会在解译至该指令时强制重新读档；若 `M198` 被 `Pr3601~3610` 登录为 M 码调用宏，开档功能无效。
- `M98/M198` 的子程序继承调用方 `#1` 至 `#400`。子程序内执行 `M02/M30` 视同结束子程序并返回主程序。
- `M99` 无引数时返回 `M98/M198` 下一行；`M99 P_` 返回主程序指定 N 序号；`M99 Q_` 返回主程序指定行号。

### 2.3 预解限制

`WAIT()` 不能被当作“等待子程序完成”的通用同步原语。官方明确列出 `M98`、`M99`、`M198` 不在它的等待保证范围内；涉及调用和运动时序的程序必须通过图形模拟或实机验证。

## 4. 后续验证设计

### 自动化测试

在 `v2.10.0` 发布后，先向现有导航测试加入以下边界：

1. `G65/G66/G66.1/M98/M198` 的数字目标、命名目标和静态字符串目标可被解析。
2. `P#1`、`P(#1+1)`、注释和普通字符串不产生导航或引用。
3. `G67`、`M99` 不被误判为文件调用。
4. 若新增 “最后一个 G 码” 诊断，分别覆盖合法、违规、注释和多 G 码单节。

## 5. CALL-07 代码审查结论

审查日期：2026-07-13。审查范围为 `src/navigationSymbols.js`、`src/navigationIndex.js`、`src/codeDocs.js`、`tests/extension.test.js` 与 `tests/integration/suite/syntecMacro.test.js`。

### 已符合资料包的实现

- 静态解析已覆盖 `G65`、`G66`、`G66.1` 的数字和静态字符串 `P` 目标，以及 `M98/M198` 的数字 `P` 目标。
- `P#变量`、表达式目标、注释和普通字符串均不会产生静态调用；这符合“不猜测运行时目标”的导航边界。
- 静态调用在导航索引中按 `G` 目标与 `O` 目标分别标准化，已有单元和 VS Code 集成测试覆盖引用和定义跳转。
- `G67` 和 `M99` 不匹配调用解析规则，不会被误判为跨文件调用。

### 明确不在静态分析中实现的项目

- `G65` 独立变量、`M98/M198` 继承变量、`G66/G66.1` 的触发频率与变量回收：须依图形模拟或控制器运行验证。
- `M198` 的强制读档及被 `Pr3601~3610` 登录后的例外：需控制器配置和档案变更实验。
- `WAIT()` 对 `M98/M99/M198` 的例外：须以运动/调用时序实验验证。

结论：导航实现没有发现需要在 `v2.10.0` 发布前修复的偏差。后续应先完成 CALL-08 至 CALL-12 的文档和测试增强，再决定是否实施 CALL-13。
